## OpenClaw Gateway 运维

### 症状（版本 2026.4.24 及之前）
- `openclaw status` / `openclaw update check` 等命令全部超时（30s+）
- `curl http://localhost:18789` 返回 connection refused
- `ps aux | grep openclaw` 显示 gateway 进程在跑但 0% CPU
- 日志 `~/.openclaw/logs/gateway.log` 最后一条是 `"starting channels and sidecars..."` 然后停住
- 日志不写，但 `kill -USR2 <pid>` 可强制刷新日志缓冲区

### 根因（版本 2026.4.24 及之前）
OpenClaw gateway 启动时每次都重新安装所有插件的 bundled runtime deps（npm 包），共 66 个包，在 `~/.openclaw/plugin-runtime-deps/` 目录下。
- 正常需 8-12 分钟
- npm ENOTEMPTY 锁冲突时可达 30 分钟+
- 日志停在 "starting..." 是因为在等插件安装完成，并非真卡住
- 日志文件 600MB+ 会导致日志轮转出问题

### 根因（版本 2026.4.25 — 2026.4.26）
**browser 插件的 CDP 初始化会卡死 gateway**（0% CPU，端口监听但不响应 HTTP）。
- 进程 PID 存在，端口 18789 监听，但所有 HTTP 请求超时
- 日志不写入（或写入极慢）
- Dev 模式（`OPENCLAW_PROFILE=dev`）因 `skipBootstrap: true` 能起来

**修复**：在 `openclaw.json` 中关闭 browser 插件：
```python
import json
with open('/Users/mac/.openclaw/openclaw.json') as f:
    d = json.load(f)
d['plugins']['entries']['browser'] = {'enabled': False}
with open('/Users/mac/.openclaw/openclaw.json', 'w') as f:
    json.dump(d, f, indent=2)
```

### 根因（版本 2026.4.27+）— 升级后完全无法启动
**升级到 2026.4.27 后 gateway 卡在 "staging bundled runtime deps" 完全无响应。**

1. **plugin-runtime-deps 安装不完整**：同时存在 4 个版本目录（2026.4.24, 2026.4.25, 2026.4.27, unknown），版本之间文件互相引用导致缺失文件（如 `chutes-oauth-CUO6Kvd3.js`）
2. **npm 网络不通**：gateway 尝试安装 8 个缺失包（@agentclientprotocol/claude-agent-acp@0.31.0, @grammyjs/*, @homebridge/ciao@^1.3.6, acpx@0.6.1, chokidar@^5.0.0, grammy@^1.42.0 等），npm registry 访问超时后一直重试
3. **所有 openclaw CLI 命令全部超时**：`doctor`, `gateway stop`, `dashboard` 等都卡死
4. **launchctl 重启后 gateway 进程存在但不监听端口**：旧进程没真正被杀掉（launchd 管理混乱）

**Gateway token 位置**：`openclaw.json` 里存的是 `"***"`（生成后存在 `~/.openclaw/identity/device-auth.json` 的 `tokens.operator.token` 字段），该 token 是 device-to-device 认证，不用于 HTTP API。

## 执行模式（在 execute_code 或 terminal 中）

推荐在 `execute_code` 中用 Python subprocess 调用：

```python
import subprocess, json

def mmx_search(query, max_results=10):
    cmd = f'mmx search query --q "{query}" --output json'
    r = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=30)
    data = json.loads(r.stdout)
    results = []
    for item in data.get('organic', [])[:max_results]:
        results.append({
            'title': item.get('title', ''),
            'link': item.get('link', ''),
            'snippet': item.get('snippet', '').replace('</p><p>', ' ').replace('<p>', '').replace('</p>', ''),
            'date': item.get('date', '')
        })
    return results
```

## 搜索多条（多关键词合并去重）

```python
def search_briefing(keywords, max_results=5, max_total=12):
    all_results = {}
    for kw in keywords:
        try:
            results = mmx_search(kw, max_results=max_results)  # 注意参数名是 max_results，不是 max_per_kw
            for r in results:
                # 用 title 做去重 key
                key = r['title'][:30]
                if key not in all_results:
                    all_results[key] = r
        except Exception as e:
            print(f"搜索 '{kw}' 失败: {e}")
    
    items = list(all_results.values())
    items.sort(key=lambda x: x.get('date', ''), reverse=True)
    return items[:max_total]
```

## 飞书 Post 消息发送（踩坑总结）

飞书 post 消息 content 格式极易报 230001 错误（invalid message content），正确格式：

```python
# 正确的 content 格式：扁平数组，每行一个 paragraph 对象
content = [
    {"tag": "text", "text": "第一行内容"},
    {"tag": "text", "text": "第二行内容"},
    {"tag": "text", "text": "第三行内容"},
]

post_content = {
    "zh_cn": {
        "title": "标题",
        "content": content  # 直接放数组对象，不要再 json.dumps
    }
}
payload = {
    "receive_id": chat_id,
    "msg_type": "post",
    "content": json.dumps(post_content, ensure_ascii=False)
}
```

关键点：
- content 是扁平数组 `[{tag: "text", text: "..."}]`，不是 `[[{...}]]`（错误格式会导致 230001）
- content 字段本身是对象，不是字符串；外层 payload.content 才是 json.dumps 后的字符串
- 报错 230001 通常是 content 格式不对，不是 JSON 字符串，或结构错误
- 先用 text 类型消息测试 chat_id 和 token 是否正确
- 推荐用 urllib.request 而非 subprocess shell 执行 curl，避免 Unicode surrogate 编码问题

## OpenClaw Gateway 运维

### 症状（版本 2026.4.24 及之前）
- `openclaw status` / `openclaw update check` 等命令全部超时（30s+）
- `curl http://localhost:18789` 返回 connection refused
- `ps aux | grep openclaw` 显示 gateway 进程在跑但 0% CPU
- 日志 `~/.openclaw/logs/gateway.log` 最后一条是 `"starting channels and sidecars..."` 然后停住
- 日志不写，但 `kill -USR2 <pid>` 可强制刷新日志缓冲区

### 根因（版本 2026.4.24 及之前）
OpenClaw gateway 启动时每次都重新安装所有插件的 bundled runtime deps（npm 包），共 66 个包，在 `~/.openclaw/plugin-runtime-deps/` 目录下。
- 正常需 8-12 分钟
- npm ENOTEMPTY 锁冲突时可达 30 分钟+
- 日志停在 "starting..." 是因为在等插件安装完成，并非真卡住
- 日志文件 600MB+ 会导致日志轮转出问题

### 根因（版本 2026.4.25 — 2026.4.26）
**browser 插件的 CDP 初始化会卡死 gateway**（0% CPU，端口监听但不响应 HTTP）。
- 进程 PID 存在，端口 18789 监听，但所有 HTTP 请求超时
- 日志不写入（或写入极慢）
- Dev 模式（`OPENCLAW_PROFILE=dev`）因 `skipBootstrap: true` 能起来

**修复**：在 `openclaw.json` 中关闭 browser 插件：
```python

with open('/Users/mac/.openclaw/openclaw.json', 'w') as f:
```

## 执行模式（在 execute_code 或 terminal 中）

推荐在 `execute_code` 中用 Python subprocess 调用：

```python
import subprocess, json

def mmx_search(query, max_results=10):
    cmd = f'mmx search query --q "{query}" --output json'
    r = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=30)
    data = json.loads(r.stdout)
    results = []
    for item in data.get('organic', [])[:max_results]:
        results.append({
            'title': item.get('title', ''),
            'link': item.get('link', ''),
            'snippet': item.get('snippet', '').replace('</p><p>', ' ').replace('<p>', '').replace('</p>', ''),
            'date': item.get('date', '')
        })
    return results
```

## 飞书 Post 消息发送（踩坑总结）

飞书 post 消息 content 格式极易报 230001 错误（invalid message content），正确格式：

```python
# 正确的 content 格式：扁平数组，每行一个 paragraph 对象
content = [
    {"tag": "text", "text": "第一行内容"},
    {"tag": "text", "text": "第二行内容"},
    {"tag": "text", "text": "第三行内容"},
]

post_content = {
    "zh_cn": {
        "title": "标题",
        "content": content  # 直接放数组对象，不要再 json.dumps
    }
}
payload = {
    "receive_id": chat_id,
    "msg_type": "post",
    "content": json.dumps(post_content, ensure_ascii=False)
}
```

关键点：
- content 是扁平数组 `[{tag: "text", text: "..."}]`，不是 `[[{...}]]`（错误格式会导致 230001）
- content 字段本身是对象，不是字符串；外层 payload.content 才是 json.dumps 后的字符串
- 报错 230001 通常是 content 格式不对，不是 JSON 字符串，或结构错误
- 先用 text 类型消息测试 chat_id 和 token 是否正确
- 推荐用 urllib.request 而非 subprocess shell 执行 curl，避免 Unicode surrogate 编码问题

```python
import subprocess, json

def mmx_search(query, max_results=10):
    cmd = f'mmx search query --q "{query}" --output json'
    r = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=30)
    data = json.loads(r.stdout)
    results = []
    for item in data.get('organic', [])[:max_results]:
        results.append({
            'title': item.get('title', ''),
            'link': item.get('link', ''),
            'snippet': item.get('snippet', '').replace('</p><p>', ' ').replace('<p>', '').replace('</p>', ''),
            'date': item.get('date', '')
        })
    return results
```

```python
    all_results = {}
    for kw in keywords:
```

```python
import subprocess, json

def mmx_search(query, max_results=10):
    cmd = f'mmx search query --q "{query}" --output json'
    r = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=30)
    data = json.loads(r.stdout)
    results = []
    for item in data.get('organic', [])[:max_results]:
        results.append({
            'title': item.get('title', ''),
            'link': item.get('link', ''),
            'snippet': item.get('snippet', '').replace('</p><p>', ' ').replace('<p>', '').replace('</p>', ''),
            'date': item.get('date', '')
        })
    return results
```

### Step 2: Markdown 转 HTML

```python
import re

def md_to_html(md_text):
    """不含 emoji 过滤（Python 3.14 regex bug 会损坏 CJK 字符）"""
    html = md_text
    html = re.sub(r'^---\n[\s\S]+?\n---\n', '', html)
    html = re.sub(r'^# (.+)$', '<h1>\\1</h1>', html, flags=re.MULTILINE)
    html = re.sub(r'^## (.+)$', '<h2>\\1</h2>', html, flags=re.MULTILINE)
    html = re.sub(r'^### (.+)$', '<h3>\\1</h3>', html, flags=re.MULTILINE)
    html = re.sub(r'^#### (.+)$', '<h4>\\1</h4>', html, flags=re.MULTILINE)
    html = re.sub(r'^---$', '<hr>', html, flags=re.MULTILINE)
    html = re.sub(r'```(?:\w*)\n([\s\S]*?)\n```', '<pre><code>\\1</code></pre>', html)
    html = re.sub(r'`([^`]+)`', '<code>\\1</code>', html)
    html = re.sub(r'\*\*(.+?)\*\*', '<strong>\\1</strong>', html)
    html = re.sub(r'\*(.+?)\*', '<em>\\1</em>', html)
    html = re.sub(r'^> (.+)$', '<p><em>\\1</em></p>', html, flags=re.MULTILINE)
    # × 符号可能导致 500，标题级别在调用处处理
    lines = html.split('\n')
    in_table = False
    new_lines = []
    for line in lines:
        if '|' in line and line.strip().startswith('|'):
            cells = [c.strip() for c in line.strip().strip('|').split('|')]
            if not in_table:
                new_lines.append('<table border="1" cellpadding="8" cellspacing="0">')
                in_table = True
            is_header = all(re.match(r'[-:]+$', c.replace(' ', '')) for c in cells)
            tag = 'th' if is_header else 'td'
            new_lines.append('<tr>' + ''.join(
                f'<{tag}>{c}</{tag}>' for c in cells
                if not re.match(r'^[-:]+$', c.replace(' ', ''))
            ) + '</tr>')
        else:
            if in_table:
                new_lines.append('</table>')
                in_table = False
            new_lines.append(line)
    if in_table:
        new_lines.append('</table>')
    return '\n'.join(new_lines)

def extract_title(md):
    """提取标题并处理特殊字符"""
    m = re.search(r'^#\s+(.+)$', md, re.MULTILINE)
    title = m.group(1).strip() if m else '无标题'
    title = title.replace('×', 'x')  # × 会导致服务端 500
    return title
```

## OpenClaw Gateway 运维

### 症状（版本 2026.4.24 及之前）
- `openclaw status` / `openclaw update check` 等命令全部超时（30s+）
- `curl http://localhost:18789` 返回 connection refused
- `ps aux | grep openclaw` 显示 gateway 进程在跑但 0% CPU
- 日志 `~/.openclaw/logs/gateway.log` 最后一条是 `"starting channels and sidecars..."` 然后停住
- 日志不写，但 `kill -USR2 <pid>` 可强制刷新日志缓冲区

### 根因（版本 2026.4.24 及之前）
OpenClaw gateway 启动时每次都重新安装所有插件的 bundled runtime deps（npm 包），共 66 个包，在 `~/.openclaw/plugin-runtime-deps/` 目录下。
- 正常需 8-12 分钟
- npm ENOTEMPTY 锁冲突时可达 30 分钟+
- 日志停在 "starting..." 是因为在等插件安装完成，并非真卡住
- 日志文件 600MB+ 会导致日志轮转出问题

### 根因（版本 2026.4.25 — 2026.4.26）
**browser 插件的 CDP 初始化会卡死 gateway**（0% CPU，端口监听但不响应 HTTP）。
- 进程 PID 存在，端口 18789 监听，但所有 HTTP 请求超时
- 日志不写入（或写入极慢）
- Dev 模式（`OPENCLAW_PROFILE=dev`）因 `skipBootstrap: true` 能起来

**修复**：在 `openclaw.json` 中关闭 browser 插件：
```python
import json
with open('/Users/mac/.openclaw/openclaw.json') as f:
    d = json.load(f)
d['plugins']['entries']['browser'] = {'enabled': False}
with open('/Users/mac/.openclaw/openclaw.json', 'w') as f:
    json.dump(d, f, indent=2)
```

### 根因（版本 2026.4.27+）— 升级后完全无法启动
**升级到 2026.4.27 后 gateway 卡在 "staging bundled runtime deps" 完全无响应。**

1. **plugin-runtime-deps 安装不完整**：同时存在 4 个版本目录（2026.4.24, 2026.4.25, 2026.4.27, unknown），版本之间文件互相引用导致缺失文件（如 `chutes-oauth-CUO6Kvd3.js`）
2. **npm 网络不通**：gateway 尝试安装 8 个缺失包（@agentclientprotocol/claude-agent-acp@0.31.0, @grammyjs/*, @homebridge/ciao@^1.3.6, acpx@0.6.1, chokidar@^5.0.0, grammy@^1.42.0 等），npm registry 访问超时后一直重试
3. **所有 openclaw CLI 命令全部超时**：`doctor`, `gateway stop`, `dashboard` 等都卡死
4. **launchctl 重启后 gateway 进程存在但不监听端口**：旧进程没真正被杀掉（launchd 管理混乱）

**Gateway token 位置**：`openclaw.json` 里存的是 `"***"`（生成后存在 `~/.openclaw/identity/device-auth.json` 的 `tokens.operator.token` 字段），该 token 是 device-to-device 认证，不用于 HTTP API。

## 执行模式（在 execute_code 或 terminal 中）

推荐在 `execute_code` 中用 Python subprocess 调用：

```python
import subprocess, json

def mmx_search(query, max_results=10):
    cmd = f'mmx search query --q "{query}" --output json'
    r = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=30)
    data = json.loads(r.stdout)
    results = []
    for item in data.get('organic', [])[:max_results]:
        results.append({
            'title': item.get('title', ''),
            'link': item.get('link', ''),
            'snippet': item.get('snippet', '').replace('</p><p>', ' ').replace('<p>', '').replace('</p>', ''),
            'date': item.get('date', '')
        })
    return results
```

```python
def search_briefing(keywords, max_results=5, max_total=12):

    for kw in keywords:
```

## 飞书 Post 消息发送（踩坑总结）

飞书 post 消息 content 格式极易报 230001 错误（invalid message content），正确格式：

```python

content = [
    {"tag": "text", "text": "第一行内容"},
    {"tag": "text", "text": "第二行内容"},

]

post_content = {
    "zh_cn": {
        "title": "标题",
        "content": content  # 直接放数组对象，不要再 json.dumps
    }

payload = {
    "receive_id": chat_id,
    "msg_type": "post",
    "content": json.dumps(post_content, ensure_ascii=False)
}
```

关键点：
- content 是扁平数组 `[{tag: "text", text: "..."}]`，不是 `[[{...}]]`（错误格式会导致 230001）
- content 字段本身是对象，不是字符串；外层 payload.content 才是 json.dumps 后的字符串
- 报错 230001 通常是 content 格式不对，不是 JSON 字符串，或结构错误
- 先用 text 类型消息测试 chat_id 和 token 是否正确
- 推荐用 urllib.request 而非 subprocess shell 执行 curl，避免 Unicode surrogate 编码问题

```python
import subprocess, json

def mmx_search(query, max_results=10):
    cmd = f'mmx search query --q "{query}" --output json'
    r = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=30)
    data = json.loads(r.stdout)
    results = []
    for item in data.get('organic', [])[:max_results]:
        results.append({
            'title': item.get('title', ''),
            'link': item.get('link', ''),
            'snippet': item.get('snippet', '').replace('</p><p>', ' ').replace('<p>', '').replace('</p>', ''),
            'date': item.get('date', '')
        })
    return results
```

```python
import subprocess, json

def mmx_search(query, max_results=10):
    cmd = f'mmx search query --q "{query}" --output json'
    r = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=30)
    data = json.loads(r.stdout)
    results = []
    for item in data.get('organic', [])[:max_results]:
        results.append({
            'title': item.get('title', ''),
            'link': item.get('link', ''),
            'snippet': item.get('snippet', '').replace('</p><p>', ' ').replace('<p>', '').replace('</p>', ''),
            'date': item.get('date', '')
        })
    return results
```

```python
def search_briefing(keywords, max_results=5, max_total=12):

    for kw in keywords:
```

## 执行模式（在 execute_code 或 terminal 中）

推荐在 `execute_code` 中用 Python subprocess 调用：

```python
import subprocess, json

def mmx_search(query, max_results=10):
    cmd = f'mmx search query --q "{query}" --output json'
    r = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=30)
    data = json.loads(r.stdout)
    results = []
    for item in data.get('organic', [])[:max_results]:
        results.append({
            'title': item.get('title', ''),
            'link': item.get('link', ''),
            'snippet': item.get('snippet', '').replace('</p><p>', ' ').replace('<p>', '').replace('</p>', ''),
            'date': item.get('date', '')
        })
    return results
```

## 搜索多条（多关键词合并去重）

```python
def search_briefing(keywords, max_results=5, max_total=12):
    all_results = {}
    for kw in keywords:
        try:
            results = mmx_search(kw, max_results=max_results)  # 注意参数名是 max_results，不是 max_per_kw
            for r in results:
                # 用 title 做去重 key
                key = r['title'][:30]
                if key not in all_results:
                    all_results[key] = r
        except Exception as e:
            print(f"搜索 '{kw}' 失败: {e}")
    
    items = list(all_results.values())
    items.sort(key=lambda x: x.get('date', ''), reverse=True)
    return items[:max_total]
```

## 飞书 Post 消息发送（踩坑总结）

飞书 post 消息 content 格式极易报 230001 错误（invalid message content），正确格式：

```python
# 正确的 content 格式：扁平数组，每行一个 paragraph 对象
content = [
    {"tag": "text", "text": "第一行内容"},
    {"tag": "text", "text": "第二行内容"},
    {"tag": "text", "text": "第三行内容"},
]

post_content = {
    "zh_cn": {
        "title": "标题",
        "content": content  # 直接放数组对象，不要再 json.dumps
    }
}
payload = {
    "receive_id": chat_id,
    "msg_type": "post",
    "content": json.dumps(post_content, ensure_ascii=False)
}
```

关键点：
- content 是扁平数组 `[{tag: "text", text: "..."}]`，不是 `[[{...}]]`（错误格式会导致 230001）
- content 字段本身是对象，不是字符串；外层 payload.content 才是 json.dumps 后的字符串
- 报错 230001 通常是 content 格式不对，不是 JSON 字符串，或结构错误
- 先用 text 类型消息测试 chat_id 和 token 是否正确
- 推荐用 urllib.request 而非 subprocess shell 执行 curl，避免 Unicode surrogate 编码问题
