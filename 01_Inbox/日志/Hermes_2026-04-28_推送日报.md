# 图文轮播视频生成流水线

## 核心流程

```
完整文案 → 分场景段落（N段，N≈TTS总长÷3.5）
→ 每段：1张专属图 + 对应TTS配音切片
→ 每张图固定展示时长 3.5 秒，不循环复用
→ 迭代两两 xfade 交叉淡化拼接
→ TTS配音串联 + 钢琴背景乐混音
→ 压缩（<30MB）→ 飞书发送（视频+N张图+文案三件套）
```

**节奏规则（2026-04 实测教训）**：
- TTS 总长：50–90 秒（用户要求）
- 每张图展示：**固定 3.5 秒**（不是10秒！）
- 所需图片数 = TTS总长 ÷ 3.5（例：80秒 → 23张图）
- **图片数必须与文案段落数一一对应**，每张图配对应段落的文字

**⚠️ 图片生成禁止并发**：mmx API有速率限制，23张并发全部报 `usage limit exceeded` 失败。
必须串行逐张生成（每张间隔3-5秒），不可用 ThreadPoolExecutor 批量并发。

### 3. 每段专属图（解决图不对文问题）
**关键规则**：每段配1张专属图（不跨段复用），每张图固定展示 3.5 秒。

### 4. TTS切分策略
TTS总时长 ÷ 图片数 = 每张图展示时间（固定 3.5 秒）。

### 5. 交叉淡化拼接
使用 `xfade=transition=dissolve`，迭代两两拼接：

```python
def xfade_pair(a, b, ad, bd, trans=1.0, out):
    offset = ad - trans
    cmd = ['ffmpeg', '-y', '-i', a, '-i', b,
           '-filter_complex', f'[0:v][1:v]xfade=transition=dissolve:duration={trans}:offset={offset:.3f}[outv]',
           '-map', '[outv]', '-an', '-c:v', 'libx264', '-preset', 'fast', '-crf', '22',
           '-t', str(ad + bd - trans), out]
    return subprocess.run(cmd, capture_output=True, text=True)
```

迭代直到合并为单个文件（约log2(N)轮，36张图需5轮）。

### 6. 飞书文件大小限制
视频文件通过 `im/v1/files` 上传限制为 **30MB**，超出则报错 `234006`。超过30MB必须压缩：

```bash

  -vf "scale=720:-2" \
  -c:a aac -b:a 128k \
  -movflags +faststart \
  output_comp.mp4
```

#### Step 1d：轮播片段 + 交叉淡化拼接
每张图 + TTS 时长生成独立 mp4，然后用迭代两两 xfade 拼接：
```bash
# 单片段
ffmpeg -y -loop 1 -i {img}.jpg -i {tts}.mp3 -t {dur}
  -vf "scale=1080:1920:force_original_aspect_ratio=decrease,pad=1080:1920:(ow-iw)/2:(oh-ih)/2"
  -af "volume=1.5" -c:v libx264 -preset fast -crf 22 -c:a aac -b:a 128k -shortest {seg}.mp4

# 两两 xfade（dissolve 交叉淡化，比 filter_complex 链更可靠）
ffmpeg -y -i {a}.mp4 -i {b}.mp4
  -filter_complex "[0:v][1:v]xfade=transition=dissolve:duration=1:offset={dur_a-1}[outv]"
  -map [outv] -an -c:v libx264 -preset fast -crf 22 -t {dur_a+dur_b-1} {out}.mp4
```

#### Step 1e：混音输出
```bash
ffmpeg -y -i {combined}.mp4 -i {tts_all}.mp3 -i {piano}.mp3
  -filter_complex "[2:a]volume=0.2,afade=t=in:st=0:d=5,afade=t=out:st={total-8}:d=8[piano];[1:a][piano]amix=inputs=2:duration=first:normalize=0[mixed]"
  -map 0:v -map [mixed] -c:v libx264 -preset fast -crf 22 -c:a aac -b:a 128k -shortest {final}.mp4
```

#### Step 1f：压缩（飞书20MB限制）
```bash
ffmpeg -y -i {final}.mp4 -map 0:v -map 0:a
  -c:v libx264 -preset fast -crf 28 -vf "scale=720:-2"
  -c:a aac -b:a 48k -t {duration} {compressed}.mp4
```

### 已知坑（踩过）

1. **mmx `--output` 只支持目录**：`--output /dir/` 保存到 `/dir/{timestamp}.mp3`，不支持 `--output /dir/file.mp3`，需手动 copy
2. **mmx image 输出自动加 `_001`**：`--out-prefix foo` → `foo_001.jpg`，不是 `foo.jpg`
3. **ffmpeg `-shortest` 导致音视频被截断**：在 video+audio 合并时慎用，换成指定 `-t DURATION`
4. **字幕流需要 `-map 3:s`**：`-i srt` 后不自动烧入，需要明确 `-map 3:s -c:s mov_text`
5. **xfade offset 计算**：`offset = 前段时长 - TRANSITION`，xfade 从前段末尾往前 TRANS 秒开始
6. **钢琴配乐循环**：钢琴曲约 3 分钟，文案约 1.5-2 分钟，不需要循环，直接淡出即可
7. **飞书视频 20MB 限制**：最终视频需压缩 `ffmpeg -crf 26-28 -vf scale=720:-2 -c:a aac -b:a 64k`

# MiniMax mmx CLI — Non-Obvious Behaviors

## Image Generation

### `--output` flag is actually a directory, not a full path
`mmx speech synthesize --output /path/to/file.mp3` does NOT save to that path. The file is saved with an auto-generated name in CWD (e.g., `speech_2026-04-26-02-10-10.mp3`). The `--output` argument is misinterpreted.

**Workaround**: Use `--output .` (current directory) and read the `saved` field from JSON output, then copy/rename manually.

```bash
# ❌ Wrong: file ends up in CWD with auto-generated name
mmx speech synthesize "text" --output /tmp/my_tts.mp3
# Returns: {"saved": "speech_2026-04-26-02-10-10.mp3"}
# Actual location: ./speech_2026-04-26-02-10-10.mp3

# ✅ Correct pattern:
cmd='mmx speech synthesize "text" --voice male-qn-jingying --output .'
r=$(eval $cmd)
saved=$(echo $r | python3 -c "import sys,json; print(json.load(sys.stdin)['saved'])")
cp "$saved" /tmp/my_tts.mp3
```

Same applies to `mmx image generate --output /path/foo.jpg` — file lands in CWD as `foo_001.jpg`.

### Output filename suffix
`mmx image generate --out-prefix foo` appends `_001` to output filenames.

```bash
# Requested: foo.jpg
# Actual output: foo_001.jpg

# Fix: rename after generation
for f in *_001.jpg; do mv "$f" "${f%_001.jpg}.jpg"; done
```

### Full video production pipeline (slideshow + narration + music + video clips)
Used for 小红书 名画介绍 videos where MiniMax AI video clips (~5s) are too short for full narration (~60-180s).

**Pipeline**:
1. Split script into N scenes (paragraphs)
2. Generate TTS per scene → N audio files
3. Generate AI image per scene → N images
4. Optional: generate MiniMax video clips, insert 1-2 into specific scenes
5. Build slideshow per scene (image + TTS, duration = TTS length, fade in/out)
6. Concatenate all scenes
7. Concatenate all TTS into single narration track
8. Mix: final video + narration + background music

**Key parameters**:
- Video: 1080×1920 (9:16 vertical, 小红书 format)
- CRF: 22 (quality/size balance)
- Piano music volume: 0.15–0.2 (background, don't overpower narration)
- Narration volume: 1.5 (boost TTS if it sounds quiet)
- Fade in/out: 1s on scenes, music fade in 5s / fade out 8s
- CRF 22, libx264, preset fast

```python
import subprocess, json

def mmx_search(query, max_results=10):
    cmd = f'mmx search query --q "{query}" --output json'
    r = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=30)
    data = json.loads(r.stdout)
    results = []
    for item in data.get('organic', [])[:max_results]:
        results.append({
            'title': item.get('title', ''),
            'link': item.get('link', ''),
            'snippet': item.get('snippet', '').replace('</p><p>', ' ').replace('<p>', '').replace('</p>', ''),
            'date': item.get('date', '')
        })
    return results
```

```python
def search_briefing(keywords, max_per_kw=5, max_total=12):
    all_results = {}
    for kw in keywords:
        try:
            results = mmx_search(kw, max_per_kw)
            for r in results:
                # 用 title 做去重 key
                key = r['title'][:30]
                if key not in all_results:

        except Exception as e:

    items = list(all_results.values())
    items.sort(key=lambda x: x.get('date', ''), reverse=True)
    return items[:max_total]
```

## OpenClaw Gateway 运维

### 症状（版本 2026.4.24 及之前）
- `openclaw status` / `openclaw update check` 等命令全部超时（30s+）
- `curl http://localhost:18789` 返回 connection refused
- `ps aux | grep openclaw` 显示 gateway 进程在跑但 0% CPU
- 日志 `~/.openclaw/logs/gateway.log` 最后一条是 `"starting channels and sidecars..."` 然后停住
- 日志不写，但 `kill -USR2 <pid>` 可强制刷新日志缓冲区

### 根因（版本 2026.4.24 及之前）
OpenClaw gateway 启动时每次都重新安装所有插件的 bundled runtime deps（npm 包），共 66 个包，在 `~/.openclaw/plugin-runtime-deps/` 目录下。
- 正常需 8-12 分钟
- npm ENOTEMPTY 锁冲突时可达 30 分钟+
- 日志停在 "starting..." 是因为在等插件安装完成，并非真卡住
- 日志文件 600MB+ 会导致日志轮转出问题

### 根因（版本 2026.4.25+）
**browser 插件的 CDP 初始化会卡死 gateway**（0% CPU，端口监听但不响应 HTTP）。
- 进程 PID 存在，端口 18789 监听，但所有 HTTP 请求超时
- 日志不写入（或写入极慢）
- Dev 模式（`OPENCLAW_PROFILE=dev`）因 `skipBootstrap: true` 能起来

**修复**：在 `openclaw.json` 中关闭 browser 插件：
```python
import json
with open('/Users/mac/.openclaw/openclaw.json') as f:
    d = json.load(f)
d['plugins']['entries']['browser'] = {'enabled': False}
with open('/Users/mac/.openclaw/openclaw.json', 'w') as f:
    json.dump(d, f, indent=2)
```

**注意**：如果 browser 插件是你需要的（如 OpenClaw 网页 UI 的浏览器自动化功能），此修复不适用，需进一步排查 CDP 连接问题。

### 诊断步骤
```bash
# 1. 看进程状态
ps -p <pid> -o pid,ppid,state,%cpu,%mem,start,ucomm

# 2. 强制刷新日志看实时进度
kill -USR2 <pid>
tail -20 ~/.openclaw/logs/gateway.log

# 3. 看 /tmp/openclaw/ 下的实时日志（JSON 格式）
tail -f /tmp/openclaw/openclaw-$(date +%Y-%m-%d).log

# 4. 检查 npm 锁冲突
find ~/.openclaw/plugin-runtime-deps/ -name "*.lock" -delete
```

### 修复方案

**方案 A：清理后重启（推荐）**
```bash
# 清空错误日志（保留文件避免权限问题）
truncate -s 0 ~/.openclaw/logs/gateway.err.log

# 删除 npm 缓存锁
find ~/.openclaw/plugin-runtime-deps/ -name "*.lock" -delete
find ~/.openclaw/plugin-runtime-deps/ -name ".openclaw-npm-cache" -type d -exec rm -rf {} + 2>/dev/null

# 杀掉旧进程
pkill -9 -f "openclaw-gateway"

# 重启
launchctl bootstrap gui/$(id -u) ~/Library/LaunchAgents/ai.openclaw.gateway.plist
```

**方案 B：跳过飞书 channel 加速启动**
飞书 bot probe 超时（axios 30s × 4 accounts = 2 分钟延迟）：
```python
import json
with open('/Users/mac/.openclaw/openclaw.json') as f:
```

**方案 C：手动触发 npm 安装（减少重复安装）**
让 gateway 跑完一次后，下次重启会复用缓存。但 npm ENOTEMPTY 问题会反复出现，每次都要清锁。

## 执行模式（在 execute_code 或 terminal 中）

推荐在 `execute_code` 中用 Python subprocess 调用：

```python
import subprocess, json

def mmx_search(query, max_results=10):
    cmd = f'mmx search query --q "{query}" --output json'
    r = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=30)
    data = json.loads(r.stdout)
    results = []
    for item in data.get('organic', [])[:max_results]:
        results.append({
            'title': item.get('title', ''),
            'link': item.get('link', ''),
            'snippet': item.get('snippet', '').replace('</p><p>', ' ').replace('<p>', '').replace('</p>', ''),
            'date': item.get('date', '')
        })
    return results
```

## 搜索多条（多关键词合并去重）

```python
def search_briefing(keywords, max_per_kw=5, max_total=12):
    all_results = {}
    for kw in keywords:
        try:
            results = mmx_search(kw, max_per_kw)
            for r in results:
                # 用 title 做去重 key
                key = r['title'][:30]
                if key not in all_results:
                    all_results[key] = r
        except Exception as e:
            print(f"搜索 '{kw}' 失败: {e}")
    
    items = list(all_results.values())
    items.sort(key=lambda x: x.get('date', ''), reverse=True)
    return items[:max_total]
```

## 飞书 Post 消息发送（踩坑总结）

飞书 post 消息 content 格式极易报 230001 错误（invalid message content），正确格式：

```python
post_content = {
    "zh_cn": {
        "title": "标题",
        "content": [
            [{"tag": "text", "text": "第一行内容"}],
            [{"tag": "text", "text": "第二行内容"}],
            [{"tag": "text", "text": "第三行内容"}],
        ]
    }
}
payload = {
    "receive_id": chat_id,
    "msg_type": "post",
    "content": json.dumps(post_content)  # 必须是 json.dumps 后的字符串
}
```

关键点：
- content 是 array of arrays，每行一个内层数组
- 报错 230001 通常是 content 格式不对，不是 JSON 字符串，或结构错误
- 先用 text 类型消息测试 chat_id 和 token 是否正确

# Feishu API Image Upload

通过飞书开放平台 API 直接发送图片到会话。

## 完整流程

### Step 1: 获取 Token
```bash
# 凭证从环境变量或飞书开放平台获取
curl -s -X POST "https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal" \
  -H "Content-Type: application/json" \
  -d '{"app_id":"'${FEISHU_APP_ID}'","app_secret":"'${FEISHU_APP_SECRET}'"}'
```

### Step 2: 上传图片获取 image_key
```bash
curl -s -X POST "https://open.feishu.cn/open-apis/im/v1/images" \
  -H "Authorization: Bearer $TOKEN" \
  -F "image_type=message" \
  -F "image=@/path/to/image.png;type=image/png"
# 返回: {"code":0,"data":{"image_key":"img_v3_xxxx"}}
```

### Step 3: 发送图片消息
```bash
curl -s -X POST "https://open.feishu.cn/open-apis/im/v1/messages?receive_id_type=chat_id" \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"receive_id":"CHAT_ID","msg_type":"image","content":"{\"image_key\":\"IMAGE_KEY\"}"}'
```

## 关键注意事项

1. **APP_SECRET 在 env 里可能被截断** — 检查 session 历史文件获取完整值
2. **receive_id_type 要匹配** — chat_id 对应 chat_id，open_id 对应 open_id
3. **token 有效期 2 小时** — 超时需要重新获取
4. **image_key 不能跨图片复用** — 每张图都要单独上传

## 凭证查找

## ⚠️ multipart/form-data 构造常见错误

用 urllib 手动拼接时，禁止在非文件字段加 `filename=`，否则 Feishu 返回 400。

❌ 错误（把 file_name/image_type 当文件发送）：
```
--{boundary}\r\nContent-Disposition: form-data; name="file_name"; filename="video.mp4"\r\n
```

✅ 正确（用 requests 分离 files + data）：
```python
# 视频上传
files = {"file": (fname, f, "video/mp4")}
data = {"file_name": fname, "file_type": "mp4"}
r = requests.post(url, headers={"Authorization": f"Bearer {token}"}, files=files, data=data, timeout=120)

# 图片上传同理，image_type 不要放 filename 里
files = {"image": (fname, f, "image/jpeg")}
data = {"image_type": "message"}
r = requests.post(url, headers={"Authorization": f"Bearer {token}"}, files=files, data=data, timeout=30)
```

### 症状（版本 2026.4.24 及之前）
- `openclaw status` / `openclaw update check` 等命令全部超时（30s+）
- `curl http://localhost:18789` 返回 connection refused
- `ps aux | grep openclaw` 显示 gateway 进程在跑但 0% CPU

- 日志不写，但 `kill -USR2 <pid>` 可强制刷新日志缓冲区

### 根因（版本 2026.4.24 及之前）
OpenClaw gateway 启动时每次都重新安装所有插件的 bundled runtime deps（npm 包），共 66 个包，在 `~/.openclaw/plugin-runtime-deps/` 目录下。
- 正常需 8-12 分钟
- npm ENOTEMPTY 锁冲突时可达 30 分钟+
- 日志停在 "starting..." 是因为在等插件安装完成，并非真卡住
- 日志文件 600MB+ 会导致日志轮转出问题

### 根因（版本 2026.4.25+）
**browser 插件的 CDP 初始化会卡死 gateway**（0% CPU，端口监听但不响应 HTTP）。
- 进程 PID 存在，端口 18789 监听，但所有 HTTP 请求超时
- 日志不写入（或写入极慢）
- Dev 模式（`OPENCLAW_PROFILE=dev`）因 `skipBootstrap: true` 能起来

**修复**：在 `openclaw.json` 中关闭 browser 插件：
```python
import json

with open('/Users/mac/.openclaw/openclaw.json', 'w') as f:
```

**注意**：如果 browser 插件是你需要的（如 OpenClaw 网页 UI 的浏览器自动化功能），此修复不适用，需进一步排查 CDP 连接问题。

# HN Firebase API — Fresh Hacker News Data

Fetch fresh HN stories when Algolia is stale.

## The Problem

HN Algolia API (`hn.algolia.com/api/v1/search`) index can be **days to months stale**, returning old 2023-2025 stories even with recent date filters. Do NOT rely on it for current news.

## The Solution

Use the **HN Firebase REST API** — it has the freshest data:

```
GET https://hacker-news.firebaseio.com/v0/topstories.json
→ Returns array of all story IDs (newest first), e.g. [47819453, 47816960, ...]

GET https://hacker-news.firebaseio.com/v0/item/{story_id}.json
→ Returns story object: {title, url, score, descendants, time, by, ...}
```

## Fetch Pattern (Python / execute_code)

```python
import urllib.request, json, datetime, concurrent.futures

# Step 1: Get top story IDs (or use newstories.json for newest)
req = urllib.request.Request(
    'https://hacker-news.firebaseio.com/v0/topstories.json',
    headers={'User-Agent': 'Mozilla/5.0'}
)
with urllib.request.urlopen(req, timeout=30) as response:
    top_ids = json.loads(response.read().decode('utf-8'))

# Step 2: Fetch stories concurrently — sequential requests TIMEOUT at ~300s
def fetch_story(sid):
    try:
        req = urllib.request.Request(
            f'https://hacker-news.firebaseio.com/v0/item/{sid}.json',
            headers={'User-Agent': 'Mozilla/5.0'}
        )
        with urllib.request.urlopen(req, timeout=10) as r:
```

# HN Firebase API — Fresh Hacker News Data

Fetch fresh HN stories when Algolia is stale.

## The Problem

HN Algolia API (`hn.algolia.com/api/v1/search`) index can be **days to months stale**, returning old 2023-2025 stories even with recent date filters. Do NOT rely on it for current news.

## The Solution

Use the **HN Firebase REST API** — it has the freshest data:

```
GET https://hacker-news.firebaseio.com/v0/topstories.json
→ Returns array of all story IDs (newest first), e.g. [47819453, 47816960, ...]

GET https://hacker-news.firebaseio.com/v0/item/{story_id}.json
→ Returns story object: {title, url, score, descendants, time, by, ...}
```

## Fetch Pattern (Python / execute_code)

```python
import urllib.request, json, datetime, concurrent.futures

# Step 1: Get top story IDs (or use newstories.json for newest)
req = urllib.request.Request(
    'https://hacker-news.firebaseio.com/v0/topstories.json',
    headers={'User-Agent': 'Mozilla/5.0'}
)
with urllib.request.urlopen(req, timeout=30) as response:
    top_ids = json.loads(response.read().decode('utf-8'))

# Step 2: Fetch stories concurrently — sequential requests TIMEOUT at ~300s
def fetch_story(sid):
    try:
        req = urllib.request.Request(
            f'https://hacker-news.firebaseio.com/v0/item/{sid}.json',
            headers={'User-Agent': 'Mozilla/5.0'}
        )
        with urllib.request.urlopen(req, timeout=10) as r:
            item = json.loads(r.read().decode('utf-8'))
            t = item.get('title', '') or ''
            text = item.get('text', '') or ''
            url = item.get('url', '') or ''
            combined = (t + ' ' + text + ' ' + url).lower()
            # Check your keywords here
            if any(k in combined for k in YOUR_KEYWORDS):
                dt = item.get('time', 0)
                date = datetime.datetime.fromtimestamp(dt).strftime('%Y-%m-%d')
                return {
                    'title': t,
                    'url': url or f'https://news.ycombinator.com/item?id={sid}',
                    'score': item.get('score', 0),
                    'comments': item.get('descendants', 0),
                    'date': date
                }
    except:
        pass
    return None

results = []
with concurrent.futures.ThreadPoolExecutor(max_workers=20) as executor:
    futures = {executor.submit(fetch_story, sid): sid for sid in top_ids[:100]}
    for future in concurrent.futures.as_completed(futures, timeout=60):
        r = future.result()
        if r:
            results.append(r)
        if len(results) >= 20:
            break

results.sort(key=lambda x: x['score'], reverse=True)
```

**⚠️ CRITICAL: Do NOT use sequential HTTP requests** — the naive for-loop times out at 300s.
Always use `concurrent.futures.ThreadPoolExecutor` with 15-30 workers.

# HN Firebase API — Fresh Hacker News Data

Fetch fresh HN stories when Algolia is stale.

## The Problem

HN Algolia API (`hn.algolia.com/api/v1/search`) index can be **days to months stale**, returning old 2023-2025 stories even with recent date filters. Do NOT rely on it for current news.

## The Solution

Use the **HN Firebase REST API** — it has the freshest data:

```
GET https://hacker-news.firebaseio.com/v0/topstories.json
→ Returns array of all story IDs (newest first), e.g. [47819453, 47816960, ...]

GET https://hacker-news.firebaseio.com/v0/item/{story_id}.json
→ Returns story object: {title, url, score, descendants, time, by, ...}
```

## Fetch Pattern (Python / execute_code)

```python
import urllib.request, json, datetime, concurrent.futures

# Step 1: Get top story IDs (or use newstories.json for newest)
req = urllib.request.Request(
    'https://hacker-news.firebaseio.com/v0/topstories.json',
    headers={'User-Agent': 'Mozilla/5.0'}
)
with urllib.request.urlopen(req, timeout=30) as response:
    top_ids = json.loads(response.read().decode('utf-8'))

# Step 2: Fetch stories concurrently — sequential requests TIMEOUT at ~300s
def fetch_story(sid):
    try:
        req = urllib.request.Request(
            f'https://hacker-news.firebaseio.com/v0/item/{sid}.json',
            headers={'User-Agent': 'Mozilla/5.0'}
        )
        with urllib.request.urlopen(req, timeout=10) as r:
            item = json.loads(r.read().decode('utf-8'))
            t = item.get('title', '') or ''
            text = item.get('text', '') or ''
            url = item.get('url', '') or ''
            combined = (t + ' ' + text + ' ' + url).lower()
            # Check your keywords here
            if any(k in combined for k in YOUR_KEYWORDS):
                dt = item.get('time', 0)
                date = datetime.datetime.fromtimestamp(dt).strftime('%Y-%m-%d')
                return {
                    'title': t,
                    'url': url or f'https://news.ycombinator.com/item?id={sid}',
                    'score': item.get('score', 0),
                    'comments': item.get('descendants', 0),
                    'date': date
                }
    except:
        pass
    return None

results = []
with concurrent.futures.ThreadPoolExecutor(max_workers=20) as executor:
    futures = {executor.submit(fetch_story, sid): sid for sid in top_ids[:100]}
    for future in concurrent.futures.as_completed(futures, timeout=60):
        r = future.result()
        if r:
            results.append(r)
        if len(results) >= 20:
            break

results.sort(key=lambda x: x['score'], reverse=True)
```

**⚠️ CRITICAL: Do NOT use sequential HTTP requests** — the naive for-loop times out at 300s.

```python
import subprocess, json

def mmx_search(query, max_results=10):
    cmd = f'mmx search query --q "{query}" --output json'
    r = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=30)
    data = json.loads(r.stdout)
    results = []
    for item in data.get('organic', [])[:max_results]:
        results.append({
            'title': item.get('title', ''),
            'link': item.get('link', ''),
            'snippet': item.get('snippet', '').replace('</p><p>', ' ').replace('<p>', '').replace('</p>', ''),
            'date': item.get('date', '')
        })
    return results
```

```python
import re

def md_to_html(md_text):
    """不含 emoji 过滤（Python 3.14 regex bug 会损坏 CJK 字符）"""
    html = md_text
    html = re.sub(r'^---\n[\s\S]+?\n---\n', '', html)
    html = re.sub(r'^# (.+)$', '<h1>\\1</h1>', html, flags=re.MULTILINE)
    html = re.sub(r'^## (.+)$', '<h2>\\1</h2>', html, flags=re.MULTILINE)
    html = re.sub(r'^### (.+)$', '<h3>\\1</h3>', html, flags=re.MULTILINE)
    html = re.sub(r'^#### (.+)$', '<h4>\\1</h4>', html, flags=re.MULTILINE)
    html = re.sub(r'^---$', '<hr>', html, flags=re.MULTILINE)
    html = re.sub(r'```(?:\w*)\n([\s\S]*?)\n```', '<pre><code>\\1</code></pre>', html)
    html = re.sub(r'`([^`]+)`', '<code>\\1</code>', html)
    html = re.sub(r'\*\*(.+?)\*\*', '<strong>\\1</strong>', html)
    html = re.sub(r'\*(.+?)\*', '<em>\\1</em>', html)
    html = re.sub(r'^> (.+)$', '<p><em>\\1</em></p>', html, flags=re.MULTILINE)
    # × 符号可能导致 500，标题级别在调用处处理
    lines = html.split('\n')
    in_table = False
    new_lines = []
    for line in lines:
        if '|' in line and line.strip().startswith('|'):
            cells = [c.strip() for c in line.strip().strip('|').split('|')]
            if not in_table:
                new_lines.append('<table border="1" cellpadding="8" cellspacing="0">')
                in_table = True
            is_header = all(re.match(r'[-:]+$', c.replace(' ', '')) for c in cells)
            tag = 'th' if is_header else 'td'
            new_lines.append('<tr>' + ''.join(
                f'<{tag}>{c}</{tag}>' for c in cells
                if not re.match(r'^[-:]+$', c.replace(' ', ''))
            ) + '</tr>')
        else:
            if in_table:
                new_lines.append('</table>')
                in_table = False
            new_lines.append(line)
    if in_table:
        new_lines.append('</table>')
    return '\n'.join(new_lines)

def extract_title(md):
    """提取标题并处理特殊字符"""
    m = re.search(r'^#\s+(.+)$', md, re.MULTILINE)
    title = m.group(1).strip() if m else '无标题'
    title = title.replace('×', 'x')  # × 会导致服务端 500
    return title
```

## Git Push 凭证问题（HTTP/SSH 混合环境）

**问题**：git push 时一直超时或 `could not read Username`——credential helper（osxkeychain / `gh auth git-credential`）在这个环境里读不到 token，导致无法认证。

**原因**：macOS 钥匙串里的 GitHub PAT 没被 git 正确调用，HTTPS Push 无法弹出认证。

**解决方案：创建 netrc 文件**

```python
import subprocess, os
token = subprocess.run(['gh', 'auth', 'token'], capture_output=True, text=True).stdout.strip()
home = os.path.expanduser('~')
with open(f'{home}/.netrc', 'w') as f:
    f.write(f"""machine github.com
login gesilajerry
password {token}
""")
os.chmod(f'{home}/.netrc', 0o600)
```

然后在 git push 时指定使用 store helper：

```bash
GIT_TERMINAL_PROMPT=0 git -c credential.helper=store push origin main
```

**验证**：`~/.netrc` 权限必须是 600。
