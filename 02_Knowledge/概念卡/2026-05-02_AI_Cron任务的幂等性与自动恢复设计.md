# AI Cron任务的幂等性与自动恢复设计

**背景**

2026年5月1日归档的Hermes会话复盘显示，在过去48小时内发生了5次AI热点推送的idle timeout失败（凌晨04:00、上午10:00、下午14:30各一次，加上结构层提炼和半导体晨报各一次），导致约30条AI新闻未能推送。成功率从82%下降至72%，且关键问题在于：任务失败时用户并不知情，系统缺乏自动恢复机制。

**核心洞察**

Hermes Cron系统的核心漏洞：不告警的失败等于没失败。

**无人值守的陷阱**

当前Cron任务的设计隐含了一个假设："任务执行完成，用户自然会看到结果"。在成功率100%时这个假设成立，但在72%成功率时，有近30%的任务静默失败——没有飞书通知、没有错误日志推送、没有任何告警。用户不会主动来询问"今天为什么没收到AI热点推送"，而是默认"今天没有特别重要的AI新闻"。

这对于需要每日稳定输出的运营系统是致命的：内容断更是最难察觉的问题，因为它不留痕迹。

**幂等性设计的缺失**

当前Cron任务缺乏幂等性保护：同一个job_id可能被重复触发（如5月1日的`1426f276` AI热点推送16:00出现了2次执行），但没有去重机制。这意味着重复推送的风险（同一内容发两次）和失败后无恢复的风险并存。

**自动恢复的正确打开方式**

正确设计应该包含四层：

1. **快速失败+即时告警**：任务超时后的第一秒就应该触发飞书通知，而不是等到cron下次触发
2. **自动重试+幂等**：失败任务自动重试（间隔5分钟），重试时通过job_id做幂等检查（已成功过的任务不重复执行）
3. **降级策略**：复合任务（搜索+整合+推送）应可降级为"只保留已获取数据"而非完全失败
4. **状态持久化**：每次任务执行前记录状态为"执行中"，执行后更新为"成功/失败"，系统重启后可从持久化状态恢复

**对Jerry的意义**

Jerry当前运营的Hermes系统如果目标是"无人值守全自动"，当前的72%成功率是不够的。根据会话复盘的建议，需要优先处理idle timeout问题（调高`minimax provider idleTimeoutSeconds`配置），并为关键任务（AI热点推送、财经新闻推送）配置自动重试和告警机制。

这一原则适用于Jerry所有的自动化运营系统：不只是Cron任务，任何需要稳定输出的系统（财经推送、内容生成、MLCC产业追踪）都需要内建"失败即告警+自动恢复"的机制，否则运营断更将是常态。

**延伸思考**

从系统设计角度看，"快速失败+即时告警"和"自动重试+幂等"是一对相互制约的设计：如果每次失败都立即告警，重试失败后会产生大量重复告警（告警风暴）；如果重试成功后才告警，用户在任务实际失败和重试成功之间处于信息真空。

解决这一矛盾的标准做法是：引入"任务状态机"（Pending → Running → Success/Failed），告警只在状态机进入Failed且重试次数耗尽时触发，Pending和Running状态不告警。这套机制是现代运维平台（PagerDuty、Uptime Robot）的核心设计，也是Hermes系统需要补齐的。
