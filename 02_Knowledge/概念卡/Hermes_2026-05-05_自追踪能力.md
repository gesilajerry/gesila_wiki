# 自追踪能力

**背景**
Hermes 的结构层提炼任务本身也是通过 cron 触发的（job_id: 793ce3c39848，运行时间 2026-05-05 11:31:49）。这意味着对自身操作的追踪和归档也成为了被追踪的对象。

**核心洞察**
自追踪是指：系统的自我观测机制同时观测系统本身。一个 cron 任务负责把 cron 任务的输出归档，另一个 cron 任务负责把归档的结果提炼成知识卡片——而这两个任务本身又会被下一轮归档捕获。

「Hermes 输入层归档 22:30」任务（job_id: ca07fe1ef9dd）的归档内容里，包含「Hermes 结构层提炼 23:00」（job_id: 475e0d05da28）的记录，形成闭环。

**对Jerry的意义**
这种设计让系统具备「反思能力」——不是事后被动记录，而是主动将自身行为纳入知识管理循环。如果某天 Hermes 没有正常归档，次日会发现归档任务本身也进了归档文件。
避免「元任务遗漏」的关键：所有任务（包括管理任务）都走同一套 cron 框架。

**延伸思考**
自追踪的边界在哪里？如果连「哪个任务失败了」都要记录和反思，可能会产生过度工程。合理的自追踪应该只追踪「有价值保留的输出」，而不追踪「纯粹的系统操作日志」。Jerry 目前的设计中，归档层保留100%原始内容，到提炼层才做价值判断——这个分层本身已经是对自追踪边界的控制。
