# Cron 任务可靠性设计：防止关键任务失败的实践经验

**背景**
Hermes cron任务系统在2026年4月30日至5月1日期间共执行11次任务，失败2次（均为idle timeout），成功率82%。其中凌晨AI热点推送（04:00）和结构层提炼（20:45）失败，用户无感知无干预，造成内容漏采。这一问题暴露了cron任务在可靠性设计上的盲区。

**核心洞察**

Cron任务可靠性的核心不是"让每次都成功"，而是"让失败的影响可控且可恢复"。当前系统的三个盲区：

**盲区1：缺乏任务级别的失败告警**
目前cron执行状态只在日志文件中记录，用户需要主动查看才能发现失败。凌晨04:00的AI热点推送失败，用户早晨起床后才能感知，此时新闻已过时。关键任务需要独立的失败告警机制（如飞书私信通知）。

**盲区2：没有自动重试**
AI热点04:00失败后，cron系统没有在后续窗口（如05:00）自动重试。结构层提炼失败后也没有在下一小时重试。设计合理的重试策略（如指数退避：1分钟→5分钟→30分钟）可以将最终成功率提升至95%以上。

**盲区3：任务依赖链没有显式建模**
AI热点推送依赖xurl搜索+模型生成+飞书推送三个子步骤，任一步骤失败都导致整体失败，但系统只记录最终状态。应该对每个子步骤独立计数，便于快速定位瓶颈。

**可靠性设计四原则**

1. **快速失败+即时告警**：关键任务（AI热点、财经推送）失败后30秒内发送飞书告警
2. **自动重试+幂等设计**：重试时任务可重复执行而不产生副作用（如重复推送）
3. **降级策略**：主任务失败时启用简化版本（如从10条新闻降级为5条）
4. **状态持久化**：中间结果写入文件，重试时直接读取，避免重复网络调用

**对Jerry的意义**

Jerry的信息获取系统以"无人值守"为核心价值假设——凌晨4点的新闻、晚上的结构层提炼都应该在无人工干预的情况下完成。当前系统在这一假设上存在漏洞：任务失败了但用户不知道，而且没有自动恢复机制。补齐告警和重试后，系统才能真正做到"设置后忘记"。

**延伸思考**

可参考AWS Step Functions的状态机设计：每个任务有明确的输入、输出、成功状态、失败状态和重试策略。OpenClaw的cron系统目前较扁平，未来可在任务描述中增加`retryPolicy`和`fallback`字段，实现声明式的可靠性配置。
