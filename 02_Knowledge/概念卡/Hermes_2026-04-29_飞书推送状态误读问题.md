# 飞书推送状态误读：fallbackUsed 掩盖真实失败

**背景**
Hermes 的 Feishu 推送任务有两套状态机制：(1) 任务执行状态（cron run JSON 中的 `status` 字段）和 (2) 消息投递状态（`delivery` 对象中的 `delivered` 和 `fallbackUsed` 字段）。2026-04-29 日志显示：多个任务 `status: error`（LLM idle timeout），但 `delivery.delivered: true`，且 `fallbackUsed: true`。此时任务实际内容没有生成，但 OpenClaw 的 fallback 机制仍然向飞书发送了占位或旧消息，导致用户以为"推送成功"，实则内容空白。

**核心洞察**

问题的根本在于：当 LLM 响应超时（idle timeout）时，任务 body 没有生成内容，但 `send_message` 工具仍然调用了飞书 API，此时飞书收到的是一条错误提示或空消息，而不是 cron 设计者预期的"摘要"。系统把"飞书 API 调用成功"标记为 `delivered: true`，但这条消息对用户毫无价值。

正确的可靠性判断应使用 `fallbackUsed === false && delivered === true` 作为"真正成功"的标准。2026-04-29 日志中 MLCC 晨报和 AI 热点推送 16:00 都属于此类：summary 显示"已推送至飞书"，但 `fallbackUsed: true` 意味着主路径失败、fallback 补发，内容质量无法保证。

这类问题的本质是"投递成功 ≠ 内容有效"。在 cron 监控大盘中，需要同时拉取 `delivery.fallbackUsed` 字段，若为 `true` 则应触发告警并补发原任务，而不是信任 `delivered: true` 就放行。

另一个隐患：部分 idle timeout 任务的 `input_tokens` 约 19,000，`output_tokens` 仅 89-253，说明模型收到了极短的 prompt 或根本没收到有效 prompt 就超时了。这不是模型问题，更可能是任务配置中 prompt 模板引用了未加载的变量导致解析失败。

**对Jerry的意义**

他的 cron 系统目前以 `delivered: true` 作为任务成功的唯一判据，会漏报大量"内容空白但API送达"的情况。长期下去会形成"推送正常"的虚假安全感，直到有一天用户反馈"今天没收到新闻"才发现问题。建议在飞书推送任务的 finish 判断中增加 `!fallbackUsed` 校验。

**延伸思考**

这个模式也适用于其他 IM 渠道（Slack、Discord、微信）。任何带有 fallback 机制的消息系统在故障时都会"假性成功"——API 返回 200 但内容不是用户想要的。监控仪表盘应该同时展示主路径成功率（`fallbackUsed === false`）和 fallback 触发率作为健康指标。
