# Cron任务状态机设计与静默失败防护策略

**背景**
Memory Dreaming Promotion任务连续3次返回`status: ok` + `summary: "NO_REPLY"` + `delivered: false`，没有任何有效输出，但系统状态显示"成功"。这是一个典型的状态机设计陷阱——任务执行者（LLM）认为工作完成，但业务结果为空。理解这一陷阱对Jerry的Cron任务健康管理至关重要。

**方法论框架**

**一、Cron任务状态机的四种状态组合**

| status | summary | delivered | 真实含义 | 危险等级 |
|--------|---------|-----------|---------|---------|
| ok | 有效内容 | true | 完全成功 | ✅ 安全 |
| ok | NO_REPLY | false | 静默失败 | 🔴 高 |
| error | 有错误信息 | true | 显式失败 | ⚠️ 低 |
| error | 有错误信息 | false | 显式失败未通知 | 🔴 高 |

**二、静默失败的三个根因**

1. **任务内容为空** — Cron触发时prompt内容无法驱动有效推理
2. **极短token输入导致空响应** — 模型在输入不足时直接返回空
3. **共享资源竞争** — 多任务并行时某任务被饿死

**三、防静默失败的三道防线**

**防线1：业务结果校验（必须）**
```python
# 推送任务执行后必须检查
if push_count == 0:
    trigger_alert("静默失败：推送数量为0")
    retry_or_escalate()
```

**防线2：心跳检测（推荐）**
```
Cron任务启动 → 写入start timestamp
  ↓
每60秒检查一次进度（进度标志位）
  ↓
超时未更新 → 触发重启
```

**防线3：超时兜底（可选）**
```
Provider层 timeoutSeconds 应比 Cron层 cronTimeout 短
（确保Provider超时后还有时间重试，而非直接cron timeout）
```

**四、NO_REPLY的特殊处理**

当`summary == "NO_REPLY"`时，意味着：
- LLM没有输出任何有效内容
- 任务状态显示ok，但实际业务未完成
- delivered=false确认未有任何推送

处理策略：
1. 立即触发重试（retry）
2. 若重试仍为NO_REPLY，进入人工排查队列
3. 记录到专门的"静默失败日志"，每周复盘

**对Jerry的意义**
这套框架可用于完善Cron任务健康管理四层监控体系中的"业务结果监控"层。特别需要关注Memory Dreaming Promotion任务的设计，确认是prompt问题还是资源问题。

**延伸思考**
可以将这套框架写成自动化脚本：监控所有Cron任务，当发现status:ok + summary包含"NO_REPLY"或为空时，自动触发重试并告警。
