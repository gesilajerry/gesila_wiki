# Cron 任务管道可靠性设计

**背景**
2026-04-28 至 04-29 归档的 20 个 cron 任务中，出现过多种故障模式：日期格式 bug、msg_type 错误、内容未就绪导致空发布、网络超时等。这些问题的共同点是：管道的某个环节失效导致最终推送失败或推送了错误内容。

**核心洞察**
Cron 任务管道的可靠性需要在三个层面同时保障：

### 1. 输入层：内容就绪检测
```
小红书预生成(06:00) → 检查AI图数量>0 → 才允许发送任务(08:00/14:00/20:00)
```
未就绪时发送任务应跳过（[SILENT]），而不是发送空内容或错误内容。

### 2. 执行层：幂等 + 超时控制
- 幂等：相同内容的重复推送应可识别（如日期+批次号去重）
- 超时：每步设置合理 timeout，失败时重试或降级
- 飞书 API 调用：先发 text 测试 chat_id，再发正式内容

### 3. 推送层：格式校验
- msg_type 必须与实际内容匹配（视频→file，图片→image，文字→text）
- post 消息 content 结构必须符合 array of arrays 格式
- 图片上传前检查 5MB 红线

**对Jerry的意义**
新上线的 cron 任务应按此三层面检查清单设计；现有任务出现反复失败时，优先从"输入层就绪检测"入手（如 14:00 发送任务内容未就绪）。

**延伸思考**
- 与"并行冗余设计"（2026-04-27）结合：同一任务可设置多个执行窗口（如 08:00 + 08:05 两次），第二次检查内容是否已就绪
- OpenClaw Gateway cron 与 Hermes cron 并存时，避免重复执行：迁移时暂停 OpenClaw 任务（`cronjob pause <job_id>`）

**标签**：Hermes × Cron × 可靠性 × 飞书 × 小红书
