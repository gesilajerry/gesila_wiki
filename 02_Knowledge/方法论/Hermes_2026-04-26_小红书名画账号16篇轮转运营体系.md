# 小红书名画账号"坨坨蔑夫斯基"16篇轮转运营体系

**背景**
2026-04-26当天，小红书账号"坨坨蔑夫斯基"完成了首次完整的多篇内容运营：从06:00预生成到20:00最后发布，全天3篇内容（梵高·星月夜/莫奈·日出·印象/毕加索·格尔尼卡）顺利流转。这是首次验证整套"预生成+定时发送"架构的实战记录。

**核心洞察**

**1. 16篇轮转机制：永不重复的选题管理**
- 16幅固定名画构成选题库，按编号1-16顺序轮转，到16后重置为1
- publish_counter.txt 维护当前篇号，每次执行+1
- 每篇都有缓存原图（~/.hermes/cache/images/），绝不依赖网络下载
- 画家涵盖：梵高、莫奈、毕加索、达利、林风眠、徐悲鸿，主题偏"救赎与对艺术的向往"

**2. 三段式发布时间轴**
| 时间 | 任务 | 类型 | 关键文件 |
|------|------|------|----------|
| 06:00 | 预生成 | 生成全部3篇的文案+AI图+音乐储备 | manifest.json |
| 07:00 | 视频预生成 | 为08:00篇生成视频片段 | Hailuo-2.3-Fast |
| 08:00 | 发布 | 发送08:00篇（视频+图+文）到飞书 | mmx video |
| 10:00 | 视频预生成 | 为20:00篇生成视频片段 | Hailuo-2.3 |
| 14:00 | 发布 | 发送14:00篇（图文）到飞书 | 原图+3AI图 |
| 20:00 | 发布 | 发送20:00篇（文案）到飞书 | 视频文件缺失则降级图文 |

**3. manifest.json：内容状态追踪中枢**
每篇的状态字段（script_ready / images_ready / video_ready / ready / pending）让cron任务链得以接力——每个任务读manifest、干活、写回，无需共享内存。

**4. 音乐储备与视频片段的额度管理**
- 06:00预生成时批量生成25首印象派钢琴曲（Music-2.6），每周175首储备
- 视频：Hailuo-2.3-Fast（每日2个）+ Hailuo-2.3（每日2个）= 每天4个片段额度
- 视频6秒/片段，用ffmpeg拼接可扩展为30秒长视频

**5. 飞书发送顺序**
原图 → AI图1 → AI图2 → AI图3 → 文案（纯文本，无任何说明前缀）

**对Jerry的意义**
这套体系意味着：一个人可以完成一个小型内容工厂的自动化运营。关键不是"每天3篇"的数量，而是"预生成+接力"使得发布时间与创作时间解耦——凌晨6点一次性生成全天候内容，白天只是发送。

**延伸思考**
当前架构依赖飞书作为"发布前的中转站"——内容先发到用户飞书，用户再转发到小红书。下一步自然延伸是：直接通过小红书MCP API发布，实现真正的全自动运营。当前架构中14:00篇（图文）和20:00篇（视频/图文降级）的发布链路还未完全打通。
