# 飞书定时推送管道

**背景**
Hermes系统通过飞书IM API实现了多条定时推送管道，包括财经新闻简报（每6小时）、AI热点简报（每日4次）、深圳保障房检查（每4小时）等。这些管道在2026-04-27当日共推送18条消息，全部成功送达（code:0）。

**核心洞察**
飞书定时推送管道的架构分为三层：**触发层**（Hermes cron调度）、**内容层**（任务执行，生成简报）、**送达层**（飞书IM API）。

**送达层核心机制：**

飞书IM API发送消息的endpoint为：
```
POST https://open.feishu.cn/open-apis/im/v1/messages?receive_id_type=chat_id
```
请求体包含消息类型（text/post/image等）和接收人标识。关键参数`receive_id_type`必须设置为`chat_id`，因为我们使用的是会话ID（oc_开头）而非用户open_id。

认证采用tenant access token方式，通过：
```
POST https://open.feishu.cn/open-apis/auth/v3/tenant_access_token/internal
```
传递APP_ID和APP_SECRET获取token，有效期2小时，需在每次请求时重新获取或缓存复用。

**管道可靠性设计：**

当日18条推送100%成功，说明当前管道已经具备良好的可靠性。关键设计点包括：

第一，**token缓存复用**。APP_ID和APP_SECRET固定，但token有效期仅2小时。每次发送消息都重新请求token会产生不必要的API调用开销。可以实现一个简单的内存缓存：在token过期前5分钟刷新，复用给同一进程内的所有发送请求。

第二，**幂等发送**。飞书IM API本身是幂等的——相同的chat_id和message_id（如果传）发送相同内容不会重复创建消息。但需要注意的是，如果发送失败（网络超时、token过期等），重试时飞书会返回错误而不是重复消息。重试机制需要区分"可重试错误"（如token过期、网络抖动）和"不可重试错误"（如chat_id不存在、内容违规）。

第三，**消息格式选择**。当前使用text类型发送纯文本简报。飞书支持更丰富的post类型（支持加粗、颜色、@成员等）。如果需要更好的可读性，可以将简报升级为post类型。但要注意post类型的JSON结构更复杂，需要在任务prompt中生成正确格式的JSON。

**与wiki归档的协同：**

归档任务本身也会发送飞书报告，这意味着归档任务也在使用同一个推送管道。飞书推送失败不会影响归档文件本身（文件已写入本地磁盘），但会导致"归档完成"的确认通知丢失。这种"关键任务失败才告警，非关键任务静默失败"的策略是合理的。

**对Jerry的意义**
飞书推送管道是Hermes系统的核心输出通道。理解其认证机制和可靠性设计，有助于优化未来的消息推送任务（如增加post富媒体格式支持）和排查推送失败问题（如token过期是常见原因）。

**延伸思考**
当前推送管道是"推"模式——Hermes主动调用API发送到飞书。飞书也支持"事件订阅"模式——当用户在飞书中@机器人或发送特定关键词时，触发Hermes处理。这种"拉"模式可以实现双向交互（如用户说"查看今日财经简报"，Hermes实时生成并回复）。建议在现有管道成熟后，扩展双向交互能力，提升人机协作效率。
