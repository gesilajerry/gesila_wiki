# 搜索工具降级应急手册：建立多层次信息获取体系

## 背景
2026-04-16至17日，xreach、Tavily、Baidu、Bing、curl/RSS等外部搜索工具相继出现超时、返回乱码或服务不可用的情况。这不是偶发的网络抖动，而是**工具链单点脆弱性**的典型案例。必须建立冗余搜索体系。

## 方法论：三层搜索冗余架构

### 第一层：主力搜索（Primary）
- **xreach**（Twitter/X）：实时性强，适合科技、财经、地缘政治
- **Tavily**：深度网页搜索，适合行业报告、新闻
- **触发条件**：正常情况下默认使用

### 第二层：备用搜索（Secondary）
- **SerpApi**（刚刚配置）：Google/Bing等搜索引擎结果，$5/1000次
- **RSS feeds**：36kr、财经网、 Semiconductor Digest等垂直媒体
- **触发条件**：xreach/Tavily 超时或返回数据质量差时切换

### 第三层：应急兜底（Tertiary）
- **Jina Reader**：抓取任意URL的内容
- **触发条件**：前两层均不可用时，用Jina直接抓取已知来源URL

## 降级决策流程

```
工具调用失败？
  │
  ├─ 超时 → 立即切换第二层
  │        └─ 第二层成功 → 继续
  │        └─ 第二层失败 → 降级第三层
  │
  └─ 返回乱码/空数据 → 立即切换第二层
                          └─ 第二层成功 → 继续
                          └─ 第二层失败 → 降级第三层
                              └─ 第三层失败 → 如实告知用户"无法获取实时数据"
```

## 关键原则
1. **永远不要在工具失败时硬撑**：及时降级比浪费时间更重要
2. **永远如实告知**：搜索不到就直说，不编数据
3. **降级时保留证据**：记录哪些工具在什么时间失败，用于后续分析工具稳定性

## 触发条件检查清单（每次搜索前）
- [ ] xreach 可用？→ 试用一个简单查询验证
- [ ] Tavily 可用？→ 检查API响应时间
- [ ] SerpApi 余额充足？→ 确认credits remaining
- [ ] 已知来源RSS是否可访问？→ 检查 Feed 源状态
