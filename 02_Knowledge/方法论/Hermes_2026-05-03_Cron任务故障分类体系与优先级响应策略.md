# Cron任务故障分类体系与优先级响应策略

**背景**
2026-05-02至05-03窗口内，17次Cron任务触发产生35次Run（含重试），最终3次完全失败。失败模式分为5种：idle timeout（76%）、xreach网络分区（6%）、xreach未配置（6%）、cron timeout（6%）、NO_REPLY静默失败（18%）。不同类型的故障需要不同的响应策略。

**核心洞察**

**故障分类矩阵**

| 故障类型 | 特征 | 自动恢复能力 | 响应策略 |
|---------|------|------------|---------|
| idle timeout | 模型未在时间内输出 | 高（重试有效77%） | 自动重试，3次后升级 |
| xreach网络分区 | X.com连接超时 | 中（切换备用源可解决） | 切换BBC/Google News |
| xreach未配置 | 工具不可用 | 低（配置问题） | 人工修复配置 |
| cron timeout | 600s硬性上限 | 低（资源不足） | 精简prompt或分段执行 |
| NO_REPLY静默 | status:ok但无输出 | 无 | 人工介入任务设计 |

**优先级排序：修复成本 vs 影响范围**

```
P0 - 静默失败（NO_REPLY）：零产出+无告警，必须立即修复
P1 - idle timeout系统性：影响所有Cron任务，Provider配置修复
P2 - cron timeout：结构性限制，需重新设计任务
P3 - xreach未配置：配置不一致，单任务修复
P4 - xreach网络分区：网络问题，等待恢复或备用源切换
```

**三层响应自动化设计**

1. **即时层（秒级）**：idle timeout → 自动重试（OpenClaw内置）
2. **短时层（分钟级）**：xreach失败 → 自动切换备用源（BBC RSS）
3. **长时层（小时/天级）**：P0/P1问题 → 人工告警 + 记录到知识库

**对Jerry的意义**
当前系统的问题是过度依赖"重试"作为万能解法，而没有针对不同故障类型的专项响应。理想的设计是：idle timeout自动重试+xreach失败自动切换源+静默失败人工介入，三条线路并行，比单一重试机制可靠得多。

**延伸思考**
可以设计"Cron任务健康度仪表盘"，实时显示：
- 各任务今日成功率（目标>90%）
- 各故障类型占比趋势
- 备用源切换次数

这比事后复盘更有价值，能在故障发生时就发现趋势性异常。

**标签**：Hermes × Cron运维 × 故障分类 × 可靠性设计
