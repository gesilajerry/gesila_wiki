# Cron任务调度设计模式：从日志提炼的实战经验

**背景**
2026-04-26共执行了16个cron任务，涉及9个不同任务类型。从日志中可以看到任务编排的巧妙设计，以及一些值得记录的调度问题（如14:00任务重复执行）。

**核心洞察**

**1. 预生成-发布分离模式**
```
06:00 预生成任务 → 生成全天3篇内容的文案+图片+音乐
07:00/10:00 视频预生成 → 为08:00/20:00篇生成视频片段
08:00/14:00/20:00 发布任务 → 将预生成内容发送到飞书
```
预生成与发布分离，使得创作时间（凌晨）与用户消费时间（白天）解耦。

**2. 任务编排时间轴**
```
00:05  财经简报
01:35  AI简报
06:04  财经简报
06:18  小红书预生成（3篇内容预生成）
07:04  08:00篇视频生成
07:34  AI简报
08:04  小红书08:00发布
09:07  深圳保障房检查
10:06  20:00篇视频生成
12:03  财经简报
13:32  AI简报
14:02  小红书14:00发送（触发两次：14:02+14:27）
18:07  财经简报
19:32  AI简报
20:02  小红书20:00发送
```

**3. 调度周期的选择逻辑**
- 财经简报：每6小时一次（`0 */6 * * *`）— 覆盖早上/中午/傍晚/午夜
- AI简报：每6小时一次但偏移30分钟（`30 1,7,13,19 * * *`）— 与财经简报错开，避免同时占用资源
- 小红书发布时间：08:00/14:00/20:00 — 模拟真人活跃时间
- 视频预生成：比发布早1-2小时 — 确保发布时视频已就绪

**4. 重复执行问题（14:00任务）**
当天14:00任务执行了两次（14:02和14:27），job_id相同。这是调度系统重试机制导致的。虽然第二次执行正确识别"视频不存在，跳过"并返回[SILENT]，但说明任务本身需要幂等设计（重复执行结果一致）。

**5. 状态传递：manifest.json作为跨任务状态共享**
每个小红书任务通过读写manifest.json传递状态，而非依赖共享内存或数据库。这是一种轻量级的任务间通信方式。

**6. [SILENT]的正确使用**
当14:00任务发现视频不存在时，正确返回[SILENT]而非发送无意义的"跳过"消息。但07:34的AI简报任务也返回了[SILENT]——说明有时候"没有值得推送的新闻"也是合理的判断。

**对Jerry的意义**
Cron任务的设计原则：预生成与发布分离、错峰调度、幂等性、状态外置。这套模式可以直接套用到任何"定时内容生成+推送"的场景。

**延伸思考**
当前小红书14:00篇（图文）和20:00篇（视频/降级图文）的发布流程尚未完全自动化打通。manifest.json中记录了script_path但14:00发布任务没有读取它——这意味着20:00篇的文案可能需要手动整理。下一步应让14:00/20:00任务严格从manifest读取文案路径。
